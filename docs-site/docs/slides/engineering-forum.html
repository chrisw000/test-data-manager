The problem

Everyone hand-rolls test data — and it rots

seed_test_data_v14_FINAL.sql

4,000 lines of INSERTs. Nobody knows which tests still need which rows. Schema moved on; the script didn't.

restore-golden-db.ps1

A shared "golden" snapshot that drifts weekly. Two teams fight over its state; a restore breaks everyone's in-flight work.

prod-copy-sanitised(2).bak

A copy of production "just this once". GDPR risk, stale in a month, and the sanitisation is nobody's job.

The data a test needs is a business statement — but today it is expressed as infrastructure.

What TDM is

The business statement is the seeding instruction

@seed:42
Feature: Orders regression seed
  Scenario: Customer places an order
    Given a Customer exists with name "Acme Ltd" and tier "Gold"
    And an Order exists for Customer "Acme Ltd" with status "Pending"
    Then 1 Order should exist with status "Pending"
  • Plain Gherkin, matched by TDM's grammar — no SQL, no builders, no fixtures
  • Domain teams ship the EF Core DbContext + repositories they already have, as plugins
  • Rows land in your domain's own model, deterministic and audited
Takeaway: this six-line feature is a real, executed artifact from the repo — not pseudocode. Fallback: the same feature is on the docs home page; open it there. Docs: Start here → Getting started (runs this in ≤15 min).

Natural keys — the concept

"Acme Ltd" is the name your test means

  • Every entity has a natural key — the business identifier (Name, OrderNumber, Sku)
  • Steps reference by natural key; TDM resolves it wherever the row lives — this scenario, an earlier run, another domain
  • No surrogate-id bookkeeping in tests, ever
Takeaway: tests speak business identifiers; TDM does the lookup. This slide sets up the identity contract — the next one. Fallback: static diagram, nothing to fail. Docs: Start here → Concepts (natural keys feed the identity contract).

Editor support

Typos die in the editor, not at runtime

  • Squiggles for unknown entities/properties in <1s — validated against your resolved schema (tdm export-model)
  • Completion and hover from the same model; no database connection
  • One dotnet tool (tdm lsp): VS Code extension in-repo, works with any LSP editor
Takeaway: the #1 friction (typos becoming unmatched steps at runtime) is gone. Fallback: the mock-up mirrors the real extension; demo lives in editors/vscode. Docs: Guides → Editor setup.
CI operation

tdm validate: a gate that needs no database

  • Grammar, entity resolution and policy gates checked before any data exists
  • SARIF → inline PR annotations · JUnit → the CI test tab · one composite GitHub Action
  • Exit codes 0/1/2 map to pass/warn/fail — pipelines need no parsing
Takeaway: validate persists nothing — that's its whole appeal. This repo runs it on every PR. Fallback: show the repo's own Actions tab. Docs: Guides → CI: validate, report, gate.
Reporting & audit

Every run leaves evidence

Reference lineage from the repo's real cross-domain scenario — who points at whom, resolved from where.

  • The living HTML report: run header, scenario drill-down, lineage graph, benchmark charts, trend sparklines — one self-contained file, opens from file://
  • Audit posture: SHA-256 checksum always; optional detached signature; attribution (runner, git SHA, settings hash) in every manifest
  • Attestation: a per-run statement that no production data was used
Takeaway: the report you're seeing this deck's posture borrowed from — single file, zero network. Fallback: open output/*.html from any repo run. Docs: Reference → Reports & the manifest.
Statistical generation

Perf data that looks like prod — without being prod

"Order": {
  "properties": {
    "Status": { "weights": {
      "Pending": 0.6, "Shipped": 0.3,
      "Cancelled": 0.1 } },
    "Total": { "distribution": "lognormal",
               "mean": 120, "sigma": 1.2 }
} }

Declarative config, applied over the faker — every draw from the scenario seed. Step overrides still win.

  • 10k orders land within ~2% of declared weights — and identically on the same seed
  • Correlated datasets keep city ↔ postcode ↔ country consistent per row
Takeaway: realistic shape and reproducibility are not a trade-off here. Fallback: the distribution playground (docs, W5-P4) shows this live with sliders. Docs: Guides → Statistical generation; Reference → Configuration → entities.
